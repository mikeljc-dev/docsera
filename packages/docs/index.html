Docsera documentation

Everything you need to install, configure and run Docsera on your own server.

💬 This site runs Docsera itself — try the assistant in the corner. It answers from this page and cites its sources.

What is Docsera?

Docsera is an embeddable widget that answers questions about your documentation using AI, with citations to the sources. Think of Intercom's support chat — but open source (AGPL-3.0) and hosted by you.

  • Installed with one command: npx docsera asks a few questions and does the rest (or use Docker Compose directly).
  • Privacy-first: self-hosted, your data never leaves your server (except the calls to the LLM provider you choose — or none at all with Ollama).
  • LLM-agnostic: Anthropic, OpenAI, or local models via Ollama — for chat and embeddings, configured independently.
  • Answers with sources: every answer deep-links to the doc section it came from. Before hallucinating, it says "I don't know."

Under the hood it's RAG with hybrid retrieval: your docs are split into sections and stored in Postgres — as pgvector embeddings and as a full-text index. Each question runs both searches (semantic similarity plus exact-term matching, so identifiers like CHAT_MAX_DISTANCE are found verbatim) and the results are fused by Reciprocal Rank Fusion. If neither branch finds anything relevant, Docsera answers "I don't know" without even calling the LLM.

Quick install

The fastest way to get Docsera running is the CLI. Requirements: Docker (Compose is bundled with it) and Node ≥ 20.

npx docsera

That single command does the whole setup:

  1. A short wizard asks which LLM should answer (Anthropic, OpenAI or any compatible API, or Ollama — a locally running Ollama is auto-detected and offered as the default, which makes the whole stack free and 100% local), which docs to index (a URL, a sitemap.xml or a GitHub owner/repo), and which website will embed the widget.
  2. Everything else is generated: a .env with your keys and freshly generated secrets (ADMIN_TOKEN, database password), a docker-compose.yml that uses the prebuilt image from ghcr.io/mikeljc-dev/docsera (no cloning, no compiling), and a docsera.json with the CLI's own state. In an empty folder the files are written right there; anywhere else they go into ./docsera. If port 3000 is taken, the next free one is picked automatically.
  3. It launches and indexes: starts Postgres+pgvector and the server, waits until it's healthy, runs the first ingestion of the docs you pointed it at, and prints the one-line <script> snippet ready to paste into your site, plus the dashboard URL.

Day-to-day commands, from the same folder (or its parent):

CommandWhat it does
npx docseraFirst run: wizard + launch. Later runs: just launch.
npx docsera ingest [source]Re-index your docs, or index a new source (URL, sitemap.xml or owner/repo). Unchanged pages cost nothing (content-hash dedupe).
npx docsera upStart the stack again, e.g. after a reboot.
npx docsera downStop everything. Your data stays in pgdata/.

To change the configuration later (provider, keys, ALLOWED_ORIGINS…), edit the generated .env — every variable is documented in Configuration — and run npx docsera up again. Scripted installs work too: the wizard reads answers from stdin, so you can pipe them in CI.

Installation from source

Prefer to build it yourself, or want to hack on Docsera? This path clones the repo and builds the image locally instead of pulling it from GHCR.

Requirements

Docker and Docker Compose (bundled with Docker Desktop). That's it — Postgres with pgvector ships in the compose file.

Quick start

git clone https://github.com/mikeljc-dev/docsera.git
cd docsera
cp .env.example .env
# edit .env: LLM keys + ADMIN_TOKEN (openssl rand -hex 32)
docker compose up -d --build

This starts Postgres+pgvector, applies the migrations automatically and runs the server at http://localhost:3000.

Verify

curl http://localhost:3000/health
# {"status":"ok","version":"__DOCSERA_VERSION__"}

Ingest your docs

Installed with the CLI? Then npx docsera ingest <source> already does everything on this page for you (it detects whether the source is a page, a sitemap, a PDF or a GitHub repo). What follows is the underlying HTTP API — what you'll call from CI or from your own scripts.

Point Docsera at your existing documentation:

curl -X POST http://localhost:3000/ingest \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"type": "sitemap", "source": "https://yourdomain.com/sitemap.xml"}'

type can be:

  • "sitemap" — every listed page, up to 200. Sitemap indexes pointing to other sitemaps work too.
  • "url" — a single page.
  • "github" — every .md/.mdx file of a public GitHub repo ("source": "owner/repo"), with optional "branch" and "path" folder filter. Citations deep-link to the files on GitHub. Set GITHUB_TOKEN to raise the API rate limit.
  • "pdf" — a PDF at a public URL (max 20 MB). No OCR: it extracts embedded text, so scanned pages with no text layer won't be indexed. One section per page; citations deep-link to the exact page (whitepaper.pdf#page=3, the fragment PDF viewers use to jump to a page) and the title comes from the PDF's own metadata when set.
  • "markdown" — raw text, handy for CI or unpublished content. Pass a url: it's the document's identity for updates and the link answers will cite.

Re-ingesting an unchanged document costs nothing in embeddings (content-hash deduplication). Headings (or, for PDFs, pages) become anchors, so citations deep-link to the exact section.

Ingesting content you don't fully trust

Add "redactSecrets": true to the request body to mask known API keys, tokens, private keys and card numbers before they're embedded and stored:

curl -X POST http://localhost:3000/ingest \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"type": "url", "source": "https://internal-wiki.example.com/runbook", "redactSecrets": true}'

It's opt-in per ingestion, not a global setting, on purpose: the same masking that protects an internal wiki you haven't fully reviewed would break a payments-integration tutorial that needs to show a real test card (Stripe's 4242 4242 4242 4242, for instance — it passes the same validity checks a real card would). Decide per document, not once for the whole instance. It never touches emails or phone numbers: docs frequently list those on purpose as a support contact, and masking them would make the assistant worse, not more private. The response reports how many things were redacted per document (field redactions), so you can tell at a glance whether anything matched. Using the CLI? npx docsera ingest <source> --redact-secrets does the same thing.

Reindex from CI

Keep the assistant in sync with your docs automatically: call /ingest from your pipeline on every deploy. Unchanged pages are skipped for free, so this is safe to run on every merge. Example GitHub Actions workflow:

name: Reindex docs
on:
  push:
    branches: [main]

jobs:
  reindex:
    runs-on: ubuntu-latest
    steps:
      - name: Reingest documentation into Docsera
        run: |
          curl -sf -X POST https://your-docsera-server.com/ingest \
            -H "Content-Type: application/json" \
            -H "Authorization: Bearer ${{ secrets.DOCSERA_ADMIN_TOKEN }}" \
            -d '{"type": "sitemap", "source": "https://yourdomain.com/sitemap.xml"}'

Add the widget

One line on any site:

<script src="https://your-docsera-server.com/widget.js"
        data-server="https://your-docsera-server.com"></script>

Add your site's origin to ALLOWED_ORIGINS in .env.

Answers render simple Markdown — including code blocks with a copy button — and every answer shows 👍/👎 feedback buttons; the votes appear in the dashboard so you can spot answers that aren't landing.

Languages and customization

The widget ships its UI in English, Spanish, French, German and Portuguese, picked automatically from your page's <html lang> or the browser. Everything is configured with data-* attributes on the script tag:

  • data-primary — your brand color for the bubble, header and buttons (any CSS color, e.g. #4F46E5).
  • data-position — right (default) or left.
  • data-locale — force a language (en, es, fr, de, pt).
  • data-suggestions — starter questions shown as clickable chips, separated by |.
  • data-contact — a URL or mailto: offered as "Contact support" when the assistant doesn't know the answer.
  • data-heading / data-placeholder — override individual strings.
  • CHAT_NO_ANSWER_TEXT (server-side) — the assistant's no-answer phrase, in your users' language.
<script src="https://your-docsera-server.com/widget.js"
        data-server="https://your-docsera-server.com"
        data-primary="#4F46E5"
        data-position="left"
        data-locale="en"></script>

The dashboard

Open /dashboard on your server and paste your ADMIN_TOKEN. The Analytics tab shows your answer rate, 👍/👎 feedback, questions per day, the top unanswered questions (grouped, so repeated questions bubble up — the most direct signal of what's missing in your docs) and the most cited sections. The Conversations tab has the full history with answered/unanswered filters and each answer's feedback.

Live numbers come from this demo's own instance via an aggregates-only public endpoint (PUBLIC_STATS=true, off by default) — visitor questions are never exposed.

MCP server

Your Docsera instance is also an MCP server: AI coding agents (Claude Code, Cursor, Windsurf…) can connect to it and query your documentation directly while they work. The endpoint is POST /mcp (Streamable HTTP, stateless) and exposes two tools:

  • search_docs — hybrid retrieval over your indexed docs, returns the most relevant sections with their source links. No LLM call, so it's fast and free.
  • ask_docs — full RAG answer with source citations, same pipeline as the widget. Uses one LLM call.

For example, to add it to Claude Code:

claude mcp add --transport http docsera https://your-server.example.com/mcp

The endpoint shares the chat rate limits (CHAT_RATE_LIMIT and the daily caps), so exposing it publicly is as safe as exposing /chat. Try it against this demo: https://docseraserver-production.up.railway.app/mcp.

Discord bot

Your community can ask the docs without leaving Discord: a /ask slash command that answers with the same RAG pipeline as the widget, citing its sources. It runs inside the Docsera server you already deploy — no separate bot process, no new dependencies — and it shares the same rate limits as /chat, keyed by Discord user.

Setup

  1. Create an application in the Discord Developer Portal (dedicate it to Docsera: the server manages its commands).
  2. Copy three values into your .env: DISCORD_PUBLIC_KEY and DISCORD_APPLICATION_ID from General Information, and DISCORD_BOT_TOKEN from Bot → Reset Token. Restart the server — it registers the /ask command by itself on boot.
  3. Back in General Information, set Interactions Endpoint URL to https://your-server.example.com/discord/interactions. Discord validates the endpoint on save (your server must be publicly reachable and already restarted with the key).
  4. Invite it to your Discord server: Installation → Guild Install with the applications.commands scope (an install link appears on that page).

Then, in any channel: /ask question: How do I configure Ollama? The bot replies with an embed — the answer plus a Sources list linking to the exact doc sections. Questions and answers land in the dashboard like any other conversation, so unanswered Discord questions show up in your coverage analytics too.

Security notes: every interaction is verified with Discord's Ed25519 signature (invalid signatures are rejected), the endpoint doesn't exist unless DISCORD_PUBLIC_KEY is set, and the bot token is only used to register the command — answers are delivered through the interaction's own webhook.

Slack bot

The same idea for Slack: a slash command that answers with the same RAG pipeline as the widget, citing its sources. It's simpler to set up than Discord — Slack doesn't have a command-registration API, and delivering the answer doesn't need a bot token at all, just Slack's own response_url.

Setup

  1. Create an app at api.slack.com/apps (dedicate it to Docsera).
  2. Under Features → Slash Commands, create a command — /ask, or whatever name you like — with Request URL set to https://your-server.example.com/slack/commands.
  3. Copy the Signing Secret from Settings → Basic Information into your .env as SLACK_SIGNING_SECRET, and restart the server.
  4. Install the app to your workspace (Slack prompts for this once a Slash Command exists).

Then, in any channel: /ask How do I configure Ollama? Slack shows an ephemeral "Thinking…" only you can see, then the real answer — with a Sources line linking to the exact doc sections — posts visibly to the channel. Like Discord, questions and answers land in the dashboard, and share /chat's rate limits, keyed by Slack user.

Security notes: every request is verified against Slack's HMAC-SHA256 signature (invalid or stale — over 5 minutes old — signatures are rejected), and the endpoint doesn't exist unless SLACK_SIGNING_SECRET is set.

FAQ

Is Docsera free?

Yes. Docsera is open source under the AGPL-3.0 license and free to self-host, forever. You only pay for your own infrastructure and, if you choose a cloud LLM provider, its API usage — with Ollama both chat and embeddings run locally at zero cost. A managed (paid) cloud version is planned for the future, for teams that prefer not to run the infrastructure.

Does my data leave my server?

Self-hosted, your documents and conversations live in your own Postgres and never leave your server. The only external calls are to the LLM provider you configure for chat and embeddings — and if you use Ollama, there are no external calls at all. Docsera has no telemetry, no analytics, and no third-party services in between.

Which LLM providers can I use?

Anthropic, OpenAI, or any OpenAI-compatible API (Gemini in compatibility mode, Groq, Mistral, LM Studio, vLLM…) via OPENAI_BASE_URL — plus fully local models via Ollama. Chat and embeddings are configured independently, so you can mix providers.

What do I need to run it?

Just Docker (and Node ≥ 20 if you use the npx docsera installer). Postgres with pgvector ships in the compose file, migrations run automatically, and a single wizard command takes care of configuration, launch and the first ingestion. Any small VPS or container platform works — this very demo runs on free-tier hosting.

Can it hallucinate?

It's designed not to. The assistant answers only from your documentation: if nothing relevant passes the similarity threshold, it answers "I don't know" without even calling the LLM, and every answer cites the exact sections it came from, so users can verify. The dashboard also surfaces every unanswered question.

How does Docsera compare to Intercom Fin or Mintlify?

Products like Intercom Fin, Mintlify's assistant, DocsBot or kapa.ai solve the same problem as hosted services: your docs and your users' questions flow through their infrastructure, with models they manage, for a subscription. Docsera is the open-source (AGPL-3.0), self-hosted alternative: it runs on your server, your data stays in your Postgres, you choose the LLM (including fully local via Ollama), and it's free — you only pay your own infrastructure and optional LLM usage. If you want a managed, zero-ops product, the hosted options are excellent; if you want control, privacy and no vendor lock-in, that's Docsera.

Is there a hosted version?

Not yet. Today Docsera is self-hosted only; a managed cloud version with usage-based pricing is on the roadmap. If you'd use one, open an issue on GitHub and tell us — it helps prioritize.

Which languages does it support?

The widget UI ships in English, Spanish, French, German and Portuguese, auto-detected from your page or the visitor's browser. The assistant answers in the language the question was asked in, and the "I don't know" phrase is configurable so it matches your users' language.

Is the live demo rate-limited?

Yes. The assistant on this site is a public demo, so it has fair-use limits: a few questions per minute and around 25 questions per day per visitor, plus a global daily budget for the whole demo. If you hit a limit, the widget will let you know — just come back later. On your own self-hosted instance you set these limits (or disable them) with CHAT_RATE_LIMIT, CHAT_DAILY_LIMIT and CHAT_GLOBAL_DAILY_LIMIT.

How can I contribute?

Bug reports from real usage, focused PRs and concrete proposals are all welcome — see the contributing guide on GitHub. Docsera is AGPL-3.0: contributions are distributed under the same license.
